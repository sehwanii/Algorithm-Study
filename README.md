# 알고리즘 스터디 규칙 v1

## 1. 스터디 목적

- 백준과 solved.ac를 활용해 꾸준히 알고리즘 실력을 향상합니다.
- 문제 풀이뿐 아니라 코드 리뷰와 회고를 통해 실전 코딩 테스트 감각을 기릅니다.
- 치팅 없이 스스로 사고하는 습관을 만듭니다.

---

## 2. 진행 방식 개요

- 한 주차는 다음과 같이 진행합니다.
  - 문제 풀이 기간은 토요일에 시작하여 다음 주 금요일까지입니다.
  - 코드 리뷰 기간은 토요일, 일요일 이틀입니다.
- 주차 표기와 PR 제목 예시는 다음과 같습니다.
  - 주차 표기 예시: `2025-1주차`
  - PR 제목 예시: `2025-1주차-장세환`

---

## 3. 문제 선택 규칙

- 주간 목표는 매주 백준 문제 5문제를 해결하는 것입니다.
- 언어는 자유이며, 각자 사용하고 싶은 언어를 사용합니다.
- 난이도 규칙은 다음과 같습니다.
  - 자신의 solved.ac 티어보다 7단계 이상 낮은 문제는 지양합니다.
  - 단순히 개수만 채우기 위한 지나치게 쉬운 문제 선택은 피합니다.
- 유형 다양성 규칙은 다음과 같습니다.
  - 가능하면 같은 유형의 문제만으로 5문제를 채우지 않는 것을 지향합니다.
  - 예: 그리디 5문제만 풀기보다는 그리디, 그래프, DP 등 여러 유형을 섞어서 풉니다.
- 태그 확인은 다음과 같이 합니다.
  - 문제 풀이 전에는 백준 문제 태그를 보는 것을 지양합니다.
  - 풀이를 완료한 뒤 복습 단계에서 태그를 확인하는 것은 허용합니다.
- 입문자에게는 다음을 권장합니다.
  - 알고리즘이 처음인 사람은 solved.ac 클래스 문제를 먼저 푸는 것을 추천합니다.

---

## 4. 풀이 진행 규칙

1. 문제를 읽고 풀이 전략을 머릿속에 먼저 그립니다.
2. 고민 시간은 최대 20분으로 제한합니다.
   - 이 시간 동안 다음을 생각합니다.
     - 사용할 알고리즘과 자료구조를 결정합니다.
     - 예상 시간복잡도와 공간복잡도를 대략적으로 계산합니다.
3. 20분 안에 아이디어가 잡히면 구현을 시작합니다.
4. 20분을 고민했는데도 전혀 실마리가 잡히지 않는 경우에는 다음과 같이 합니다.
   - 공식 풀이 또는 다른 사람의 아이디어를 확인한 후 구현합니다.
   - 이때 커밋 이름은 반드시 아래 형식을 따릅니다.
     - `[백준문제번호]_[문제난이도]_이름_미해결`
   - 이렇게 미해결 처리한 문제는 다음 주 5문제 중 하나로 다시 선정하여, 이번에는 스스로 아이디어를 떠올려서 풉니다.

## 5. 깃허브 레포지토리 규칙

### 5.1 브랜치와 디렉터리 구조

- 기본 브랜치는 `main`입니다.
- 주차별 개인 브랜치는 다음 형식을 사용합니다.
  - 형식: `week-01/이름`
  - 예시: `week-01/장세환`
- 디렉터리 구조는 다음과 같습니다.
  - 브랜치 내 파일 위치는 `week-01/이름/문제번호.확장자` 형식을 사용합니다.
  - 예시: `week-01/장세환/1000.cpp`

### 5.2 커밋 규칙

- 문제 하나당 커밋 하나를 원칙으로 합니다.
- 커밋 이름 형식은 다음과 같습니다.
  - 정상 해결:
    - 형식: `[백준문제번호]_[문제난이도]_이름`
    - 예시: `1000_S5_장세환`
  - 아이디어를 보고 구현한 미해결 문제:
    - 형식: `[백준문제번호]_[문제난이도]_이름_미해결`
    - 예시: `1234_G4_장세환_미해결`
  - 이번 주 가장 기억에 남는 중요 문제:
    - 형식: `[백준문제번호]_[문제난이도]_이름_중요`
    - 예시: `14502_G4_장세환_중요`

### 5.3 PR 규칙

- 작업 브랜치는 각자의 `week-01/이름` 브랜치입니다.
  - 해당 브랜치에서 작업 후 `main` 브랜치로 PR을 생성합니다.
- PR 제목 형식은 다음과 같습니다.
  - 형식: `20xx-x주차-이름`
  - 예시: `2025-1주차-장세환`
- PR 내용에 포함하는 것을 권장하는 항목은 다음과 같습니다.
  - 이번 주에 푼 문제 목록
    - 예시: `1000 S5 / 1037 S4 / 14502 G4`
  - 이번 주 본인의 중요 문제와 그 이유 한두 줄
  - 이번 주에 새로 배운 점 또는 깨달은 점 한두 줄

## 6. 코드 주석과 기록 규칙

각 문제 파일에는 다음 내용을 주석으로 남깁니다.

- 문제 정보:
  - 백준 문제 번호, 제목, 난이도, 링크를 기록합니다.
- 예상 시간복잡도와 공간복잡도를 기록합니다.
- 풀이 아이디어 요약:
  - 어떤 알고리즘과 자료구조를 사용했는지 적습니다.
  - 핵심 아이디어를 한두 줄로 정리합니다.
- 시간 기록:
  - 생각 시간(고민 시간)이 몇 분이었는지 기록합니다.
  - 구현 시간(코딩 시간)이 몇 분이었는지 기록합니다.
- 오답이 있었을 경우:
  - 틀렸던 이유를 명시합니다.
    - 예시: 경계 조건 누락, 자료형 오버플로, 정렬 기준 착오 등입니다.

예시는 다음과 같습니다.

```cpp
// BOJ 14502 연구소 G4
// 링크: https://www.acmicpc.net/problem/14502
// 알고리즘: 브루트포스 + BFS입니다.
// 시간복잡도: O(N^2 × 빈칸 조합)입니다.
// 공간복잡도: O(N^2)입니다.
//
// 생각 시간: 15분입니다.
// 구현 시간: 40분입니다.
// 틀렸던 이유: 처음에 벽 세 개를 중복 선택하는 경우를 고려하지 않아서 조합 구현을 수정했습니다.
```

## 7. 노션 기록 규칙

- 사람마다 이번 주에 푼 5문제 중 가장 기억에 남는 문제 하나를 선택합니다.
- 선택한 문제는 다음과 같이 처리합니다.
  - 커밋 이름에 `_중요`를 붙여 제출합니다.
    - 예시: `[14502]_G4_장세환_중요`
  - 노션에는 다음 내용을 작성합니다.
    - 문제 링크
    - 이 문제가 왜 기억에 남는지에 대한 설명
    - 핵심 아이디어 정리
    - 다시 풀 때 주의할 점

## 8. 코드 리뷰 규칙

### 8.1 리뷰 개수

- 토요일과 일요일 이틀 동안 최소 3개 이상의 문제에 리뷰를 남깁니다.
- 이 중에서 다음 조건을 만족하도록 합니다.
  - 2개 이상은 다른 사람의 중요 문제에 대한 리뷰입니다.
  - 1개 이상은 그 외 일반 문제에 대한 리뷰입니다.

### 8.2 리뷰 방식

- 리뷰 내용에는 다음과 같은 요소를 포함하는 것을 권장합니다.
  - 좋았던 점:
    - 가독성이 좋은 부분
    - 깔끔한 아이디어, 적절한 자료구조 선택 등
  - 개선 가능성:
    - 불필요한 반복, 리팩터링 포인트, 더 나은 복잡도 가능성 등
  - 궁금한 점:
    - 특정 로직에 대한 질문
    - 다른 접근 방식에 대한 제안
- 다음과 같은 방식은 지양합니다.
  - 단순 비난이나 근거 없는 평가입니다.
  - 맥락 없이 정답 코드를 강요하는 방식입니다.

## 9. 테스트케이스와 외부 도구 사용 규칙

- 기본 원칙은 다음과 같습니다.
  - 직접 만든 테스트케이스로 최대한 검증합니다.
  - 경계값, 작은 입력, 큰 입력, 특수 케이스 등을 스스로 생각해 봅니다.
- `https://testcase.ac` 사용 규칙은 다음과 같습니다.
  - 아래 조건을 모두 만족할 때에만 사용을 고려합니다.
    - 직접 엣지 케이스를 최소 5회 이상 고민하고 작성해 보았습니다.
    - 그럼에도 어디서 틀리는지 도저히 찾기 어려운 경우입니다.
  - 가능한 한 사용을 지양하며, 마지막 수단으로만 활용합니다.
- 만약 testcase.ac로 버그를 찾았다면 다음과 같이 합니다.
  - PR 설명이나 코드 주석에 해당 케이스를 간단히 남깁니다.
  - 예시: “모든 값이 음수인 경우를 고려하지 않아 실패했습니다.” 등으로 기록합니다.

## 10. AI 및 치팅 관련 규칙

- AI 사용은 절대 금지입니다.
  - ChatGPT, Copilot 등 코드나 풀이를 생성하는 모든 AI 사용을 금지합니다.
  - 아이디어, 정답 코드, 테스트케이스를 AI에게 묻는 행위도 금지합니다.
- 치팅은 절대 금지입니다.
  - 인터넷에 공개된 정답 코드를 그대로 복사하는 행위는 금지입니다.
  - 변수명만 바꾸거나 형식만 살짝 수정하는 수준의 얕은 변형도 치팅으로 간주합니다.
- 부득이하게 참고를 한 경우에는 다음과 같이 합니다.
  - 공식 에디토리얼이나 다른 사람의 코드를 참고했다면 그 사실과 출처를 PR에 명시합니다.
  - 이후 본인 언어로 다시 설명하고, 스스로 이해한 방식으로 구현하는 것을 목표로 합니다.

## 11. 디스코드 알고리즘 스터디

- 매주 주말 디스코드에서 알고리즘 스터디를 2시간 진행합니다.
  - 정확한 시간은 추후에 지정합니다.
- 참여는 자율이지만, 가능하면 참여하는 것을 권장합니다.
- 예시 진행 방식은 다음과 같습니다.
  - 각자 이번 주 중요 문제를 하나씩 설명합니다.
  - 헷갈렸던 개념을 함께 정리합니다.
  - 다음 주에 집중할 알고리즘 유형을 논의합니다.

---

## 12. 추후에 정할 항목

- 코딩 스타일 가이드는 다음과 같이 추후에 정합니다.
  - 언어별 들여쓰기, 변수명 규칙, 파일명 규칙 등을 합의 후 문서화합니다.
- 실패했을 때 규칙은 다음과 같이 추후에 제정합니다.
  - 주간 5문제를 다 채우지 못했을 때의 처리 방안
  - 리뷰 3개를 채우지 못했을 때의 처리 방안
  - 위 상황들에 대한 구체적인 규칙은 스터디를 어느 정도 진행한 후 합의하여 제정합니다.
